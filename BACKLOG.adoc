== EPIC 2 : Plugin Gradle — Logging Automatique Colab → Sheets

**Statut** : A FAIRE
**Priorite** : P0

*Pourquoi* : Premiere brique du pipeline. Sans logs structures, pas de detection
d'erreur, pas de correction IA. Le plugin doit injecter des cellules de log dans
n'importe quel notebook `.ipynb`.

=== Sous-taches

* [ ] Definir l'extension Gradle `notebook { ... }` (spreadsheetId, sheetName,
  autoInstrument)
* [ ] Implemeter `LoggingInjector` :
  - Parser le JSON du notebook (`nbformat`)
  - Identifier les cellules de code existantes
  - Wrapper `try/except` autour de chaque etape critique
  - Injecter `log_step(phase, step, status, duration, error)` via `gspread`
* [ ] Template de cellule de log standardisee (initialisation auth +
  `log_step()`)
* [ ] Tache `instrumentNotebook` : lit un `.ipynb`, injecte les logs, ecrit la
  version instrumentee
* [ ] Tests : notebook vide → notebook instrumente, validation `nbformat`
* [ ] Documentation : `README.adoc` expliquant l'injection

=== Consommation

```kotlin
// build.gradle.kts du projet consommateur
plugins { id("com.cheroliv.notebook") version "0.1.0" }

notebook {
  spreadsheetId = "1ABC123..."
  sheetName = "Logs PlantUML Expert"
  autoInstrument = true
}
```

== EPIC 3 : Déploiement Script GAS (Google Apps Script)

**Statut** : A FAIRE
**Priorite** : P0

*Pourquoi* : Le script GAS est le pont entre Sheets et GitHub. Il detecte les
erreurs, exporte en CSV, et push vers le repo. Le plugin Gradle doit pouvoir le
deployer automatiquement sur le spreadsheet.

=== Sous-taches

* [ ] Template GAS `detectAndExport.gs` :
  - `detectErrors()` : scan des lignes post dernier export
  - `exportToCsv()` : conversion feuille → CSV
  - `pushCsvToGitHub()` : `UrlFetchApp` → API GitHub
  - `installTrigger()` : timer toutes les 10 min
* [ ] Gestion des secrets GAS via `PropertiesService` (GITHUB_TOKEN,
  GITHUB_REPO, CSV_FOLDER, BRANCH)
* [ ] Tache Gradle `deployGasScript` :
  - Deploie le script sur le spreadsheet via Google Apps Script API
  - Installe le trigger temporel
* [ ] Tests : mock `UrlFetchApp`, validation CSV export, validation push
* [ ] Documentation : schema de deploiement, secrets a configurer

=== Points de vigilance

- `onEdit` ne reagit pas aux ecritures `gspread` → timer obligatoire
- Le token GitHub doit avoir droits `contents: write`
- Rate limiting : pas plus d'un push toutes les 10 min

== EPIC 4 : CI/CD — GitHub Actions + Analyse CSV

**Statut** : A FAIRE
**Priorite** : P1

*Pourquoi* : Une fois le CSV dans Git, il faut le traiter. Le plugin fournit
une tache Gradle `analyzeColabLogs` lisible par n'importe quel workflow
GitHub Actions.

=== Sous-taches

  - Detection de degradation : duree en hausse sur 3+ runs consecutifs

* [ ] Tache Gradle `analyzeColabLogs` :

* [ ] Template GitHub Actions `colab-logs.yml` :
  - `on.push.paths → logs/colab/**.csv`
  - `./gradlew analyzeColabLogs`
  - Alerte si `failure()`
* [ ] Tests : CSV avec/sans erreurs, cas limites (CSV vide, malforme)
* [ ] Documentation : integration CI/CD, exemple workflow

== EPIC 5 : Correction IA du Notebook

**Statut** : A FAIRE
**Priorite** : P1

*Pourquoi* : Le pipeline ne sert a rien s'il detecte sans corriger. L'EPIC 5
ferme la boucle : une tache Gradle assistee par LLM (open code, Claude Code)
propose une correction, push sur branche `fix/` et met a jour Google Drive.

=== Sous-taches

* [ ] Implemeter `NotebookCorrector` :
  - Construire le prompt IA (CSV logs + notebook source + stack trace + plan
    opencode)
  - Appeler l'IA (CLI open code ou API LLM)
  - Valider le notebook corrige (`nbformat.validate()`)
  - Logger la correction proposee dans Sheets
* [ ] Tache Gradle `correctNotebookViaIA` :
  - Checkout branche `fix/colab-{date}`
  - Ecrit notebook corrige
  - Commit + push
  - Met a jour Google Drive via API
* [ ] Garde-fous :
  - Ne jamais pusher sur `main` sans PR
  - Validation nbformat obligatoire avant push
  - Si 3 corrections consecutives echouent → escalade humaine
* [ ] Fichier `PLAN.adoc` comme memoire persistante open code :
  - Decisions passees, lecons apprises, regles "a ne pas faire"
* [ ] Tests : mock LLM, validation nbformat, cas d'echec
* [ ] Documentation : flux de correction, garde-fous, exemple PLAN.adoc

=== Classification des erreurs et actions IA

[cols="1,2,2"]
|===
| Categorie | Exemple | Action IA

| Secret expire
| `HF_TOKEN` invalide
| Proposer rafraichissement + check precoce

| OOM GPU
| `CUDA out of memory`
| Reduire `MAX_SAMPLES`, ajuster `BATCH_SIZE`

| Dependance manquante
| `ModuleNotFoundError`
| Ajouter package dans cellule d'installation

| Timeout reseau
| `ConnectionError`
| Ajouter retry + delai

| Syntaxe PlantUML
| Texte hors `@startuml/@enduml`
| Ajuster prompt systeme + temperature

| Conversion GGUF
| `convert_hf_to_gguf.py` erreur
| Verifier compatibilite, proposer fallback
|===

== EPIC 6 : Modelisation Historique CSV

**Statut** : A FAIRE
**Priorite** : P2

*Pourquoi* : L'accumulation de CSV dans Git forme une serie temporelle. A terme,
on peut predire les durees, les echecs, et la degradation GPU avant de lancer
un run.

=== Sous-taches

* [ ] Implemeter `TimeSeriesModeler` :
  - Agregation multi-CSV (`logs/colab/*.csv`)
  - Regression duree par phase (RandomForest, LinearRegression)
  - Detection de tendance (LinRegress pente/semaine)
  - Classification probabilite d'echec (phase, heure, jour, samples)
* [ ] Tache Gradle `modelColabLogs` :
  - Entraine sur l'historique CSV
  - Sort un rapport : duree predite, risque d'echec, fenetre optimale
* [ ] Tests : validation croisee, metriques (MAE, precision, rappel)
* [ ] Documentation : schema des colonnes, exemples de modelisation

=== Exemples de modelisation

*Prediction de duree* : "Ce run prendra ~45 min, evite vendredi soir"
*Detection de degradation GPU* : "Les T4 ralentissent de 12s/semaine le vendredi"
*Risque d'echec* : "Phase GGUF a 78% de probabilite d'echec si > 5000 samples"

== EPIC 8 : Dashboard Spreadsheet → CI event → CD response

**Statut** : A FAIRE
**Priorite** : P0

*Pourquoi* : Le sens de la suite. Le spreadsheet n'est pas un dump passif —
c'est un dashboard operationnel. Une cellule modifiee = un evenement CI.
Un evenement CI = une tache Gradle. Une tache Gradle = une reponse CD
(notebook corrige, modele redeploye, alerte envoyee). La boucle est fermee
du spreadsheet au produit final.

=== POC — Dashboard minimum viable

Avant d'ecrire une ligne de plugin, on valide le concept avec un POC
manuel :

. *Dashboard Sheet* — une feuille dediee avec :
  - Dernier statut par notebook (OK / ERROR / RUNNING)
  - Derniere duree par phase
  - Compteur d'erreurs depuis le dernier succes
  - Cellule `TRIGGER_FIX` (bool) — mise a `TRUE` manuellement ou par GAS
. *Event CI* — un check GAS ou un poll cron qui lit `TRIGGER_FIX`
. *Response CD* — si `TRUE` :
  - `git checkout -b fix/auto-{timestamp}`
  - La tache Gradle corrige le notebook
  - `git push` + PR
  - Reset `TRIGGER_FIX` a `FALSE` + log dans Sheets

[plantuml]
----
@startuml
!theme plain
skinparam backgroundColor #FEFEFE

rectangle "Dashboard\nSpreadsheet" as dash #LightYellow {
  rectangle "Status: ERROR" as status
  rectangle "TRIGGER_FIX: TRUE" as trigger
}

rectangle "GAS Timer" as gas
rectangle "GitHub Actions" as ci
rectangle "Plugin Gradle" as gradle
rectangle "Repo Git" as repo
rectangle "Google Drive" as drive

dash -> gas : poll 5min
gas -> ci : API dispatch\n(workflow_dispatch)
ci -> gradle : analyzeColabLogs\n+ correctNotebookViaIA
gradle -> repo : push fix branch
gradle -> drive : update notebook
gradle -> dash : reset TRIGGER_FIX\n+ log resolution
@enduml
----

* [ ] *POC manuel* :
  - Creer le spreadsheet dashboard avec les colonnes/statuts ci-dessus
  - Ecrire un script GAS minimal qui lit `TRIGGER_FIX` et pingue une URL
  - Valider que le cycle complet fonctionne une fois a la main
* [ ] *Automatisation GAS* :
  - Fonction `checkDashboard()` qui scanne les statuts
  - Declaration du trigger temporel
  - Logger les decisions dans une feuille `audit_log`
* [ ] *Integration CI/CD* :
  - Workflow GitHub Actions declenche par `workflow_dispatch` (depuis GAS)
  - La tache Gradle recoit le notebook ID en parametre
* [ ] *Boucle de feedback* :
  - Apres correction, le plugin ecrit le resultat dans Sheets
  - Reset de `TRIGGER_FIX`
  - Notification (email, Slack, Discord) si succes ou echec
* [ ] *Documentation* :
  - Schema du dashboard (colonnes, formules, format conditionnel)
  - Guide de deploiement du GAS + trigger
  - Exemple complet de cycle : Colab crash → dashboard rouge → TRIGGER →
    correction → dashboard vert

=== Criteres d'acceptation

- [ ] Un spreadsheet avec statuts de notebooks mis a jour automatiquement
- [ ] Une cellule `TRIGGER_FIX` qui, mise a `TRUE` (manuel ou GAS), declenche
  le pipeline CI/correction
- [ ] Le notebook corrige apparait dans Drive et sur une branche Git
- [ ] Le dashboard repasse au vert apres correction
- [ ] Le cycle complet est documente et reproductible

=== Pourquoi c'est P0 maintenant

Le dashboard spreadsheet est le chainon manquant entre "on loggue dans Sheets"
et "on fait du CI/CD piloté par les données". C'est le MVP du pipeline
d'observabilité en boucle fermée. Sans lui, les logs sont juste des logs.
Avec lui, les logs deviennent des commandes.
